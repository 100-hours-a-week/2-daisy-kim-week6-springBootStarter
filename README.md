```bash
 ├── src/main/java/com/example/projectName

 │    ├── service

```

## 회고
3학년 떄 프로젝트 막 처음 시작해 볼 떄가 생각 났다.<br>
그때 간단한 거 파일 하나 작성하는 데 하루종일 걸리고...<br>
이거 하는데 무슨 이틀이나 걸릴 거라고는 생각도 못했다.<br>
뭔가 아무것도 안 되는 게 너무 답답했다.<br>
너무 뭐가 안 돼서 이렇게 답답함을 느낀 게 3학년 때를 생각나게 했다..<br>
GPT 닥달해서 이해하고 코드 쓰고 어찌저찌 끝냈다.<br>

그래도 하나 다행인 건 각 계층마다 공식마냥 정해져있는 느낌이라 좀 익숙해 지면 괜찮을 것 같은데 사실 또 하다보면 끝이 없을 것 같다.<br>
팀원들 보면 로그인에서 애 먹던데 좀 걱정이다.<br>
그리고 다른 분들은 그냥 커뮤니티 백엔드로 다 하는 것 같던데 엄청 막힐 걸 알아서 두 번 해 보려고 따로 레포지토리를 파서 헀는데<br>
프로젝트 어떻게 시작하는지 따로 정리하면서 하다보니 더 오래 걸린 것도 있고 스프링 부트 시작하는 거 자체는 이제 괜찮을 것 같다.<br>

과제 1번 하면서 sql 쓰는 게 작년에 jsp로 프로젝트 했던 게 생각이 났다.<br>
뭔가 할 만 하다고 생각도 하면서 계속 .env 파일 만든 게 연결이 안 돼서 그냥 properties 파일에 싸그리 쓰고 gitignore에 properties를 넣었다.<br>
배포할 때 영향을 미칠 것 같아서 이게 맞는지 모르겠지만 우선... 냅다 해버린<br>
처음에 controller, service, rpository, dto 계층만 썼다.<br>
응답이 잘 왔고 뭔가 postman도 해 보고 넘어갈까 했는데 그건 커뮤니티 백엔드 구현하면서 해 보기로 했다.<br>

저번주 spring 개념 정리하면서 entity 계층이랑 Dto 계층이랑 큰 역할 차이 없이 restful api를 위해 json으로 처리할 수 있냐 차이만 있는 줄 알았다.<br>
그래서 entity 계층은 필요 없는 줄 알았는데 2번 하면서 둘 차이를 이해할 수 있었다.<br>
datagrip 쓰면서 db도 미리 세팅해 놨기도 하고 js 할 때는 카멜 형식을 많이 써서 db 컬럼도 카멜 형식으로 했는데<br>
여기서 자꾸 users 테이블에 user_id 못 쓴다? 뭐 자꾸 컬럼 오류가 나면서 애를 먹었다.<br>
그리고 테이블 이름도 users가 아니고 User였는데 자꾸 뭐가 안 돼서 걍 버려버림<br>
근데 카멜이랑 스네이크 형식에서 충돌난 거라 하길래 그냥 db에서 테이블을 전부 지워버렸다.<br>
jpa 써서 entity로 직접 db를 제어한다는 게 꽤 새롭게 다가왔다.<br>
팀원들이 jpa 쓰면 sql문 몰라도 된다~ 훨씬 편하다~ 하는 말을 하루 끝날 때쯤 알게 되었다.<br>

그리고 인텔리제 쓰면 그냥 코드 좀만 입력해도 다 자동완성해 주는데 이게 맞나? 싶었다.<br>
그래서 다른 분들 막 물어보고 다녔는데 다들 코파일럿이나 자동완성 기능 쓰신다길래 그냥 쓰기로 했다.<br>
처음에 계층별 패키지 만들고 자바 파일 만들고 @만 입력했는데 알아서 계층에 맞는 어노테이션 제안해 주길래 너무 편하다고 생각은 했다.<br>
근데 이게 과연 실력향상에 도움이 될까? 그냥 흐름 정도만 이해하면 될까? 하는 생각도 들었다.<br>
근데 또 처음에는 왜 그 자동완성을 제안해 주는지 이해하려고 또 분석하고... 흠...<br>
암튼 이래저래 이 부분에 대해 내가 어떤 자세를 가져야 하나 하는 고민이 들었다.<br>

아 그리고 처음 프로젝트 생성할 때부터 애먹기도 했다.<br>
프로젝트를 생성하고 깃허브 레포지토리에 git add하면 gradle 연결이 안 되고<br>
레포지토리 먼저 만들고 클론한 데서 프로젝트 생성하면 다른 폴더에 생성되거나 origin에 폴더 자체로 반영돼서<br>
막상 구현한 거만 빼고 전부 커밋되고 여기에만 또 시간을 좀 썼다.<br>
그래서 그냥 프론트엔드 하던 대로 우선 클론하는 걸 먼저 해서 하자고 생각하다가<br>
프로젝트 생성은 어떻게 해결하지 했는데 모듈 생성으로 하니까 해결돼서 다행이었다.<br>
이때가 ㄹㅇ 답답함 맥스였다...<br>
무슨 시작도 안 되니까 돌아버릴 것 같은? 후~ 한숨 푹푹이었지만...<br>
한 번 해결되면 할 만 하니까...<br>
그래서 깃허브에서 revert를 처음 써봤다.<br>
이것도 또 하나 경험 생겼다는 생각에 괜찮았다.<br>
웬만해서 깃허브에서 오류가 크게 생기는 편은 아니어서 revert를 쓸 일이 없었는데<br>
깃허브 충돌 문제는 아니었지만 커밋이 일부만 되는 문제로 처음으로 돌아가니까 답답했는데 그래 한 번은 써봐야지 싶은 생각도 했다.<br>
그래서 controller, service, rpository, dto 이 코드만 한 4번 쓴 것 같다 ㅋㅎㅋㅎㅋㅎㅋㅎ....(사실 안 웃김)<br>

근데 음 뭔가 프론트엔드는 확실히 계속 변화하는 트렌드가 있을 수밖에 없겠다는 느낌을 받았다.<br>
백엔드는 역할별 규칙(?)이 명확해서 잘 안 바뀔 것 같다는 생각 때문이다.<br>
그래서 백엔드 만렙 찍으면 좀 재미 없어질 것 같다는 생각도 했다.<br>
(거의 무슨 PT 한 번 받고 김종국 되면 어떡해요? 같은 소리 하고 있음;;)<br>
그래도 api 응답 오는 꼴을 직접 보니까 재밌기도 하고 Django헀을 때도 생각났다.<br>
그리고 해커톤 때 대니얼이랑 얘기 좀 많이 하면서 했는데 그 경험이 또 이번 과제할 때 도움이 된 것 같다.<br>
그때도 api 명세서를 내 기준으로 쓰다보니 카멜 형식으로 작성했었는데 대니얼이 DB에 저장되는 거랑 api 반환해 주는 거랑 바뀔 때가 있었다.<br>
이제야 왜 그랬는지 너무 납득이 가버렸다.<br>
확실히 백엔드랑 프론트엔드랑 반환할 때 백엔드 처음하는 친구들이 막혔다고 한 부분들이 한두 개 느껴졌다.<br>

그리고 Django 했을 때가 생각난 이유는 장고 할 때, urls.py에 api uri들 전부 모아서 쓰고 serializer랑 매핑(?)해 줬어야 했는데 이거는 @GetMapping, @RequestMapping이 진짜 사기라고 생각했기 때문이다.<br>
그리고 또 스웨거로 백엔드가 구현한 거 보면 user api 몇 개 따로, board api 몇 개 따로 구분돼있었는데 이게 어떻게 가능한 거지? 헀는데 Dto 하나로 여러 개 api 만드는 거라는 게 이해됐다.<br>
물론 장고 할 때도 구분되긴 했는데 확실히 이해하고 하는 거랑 느낌이 다르달까?<br>
그리고 그 장고 했을 때 어떻게 돌아가는 건지 또 이해된 느낌?<br>
그리고 또 장고에서는 models.py에 쓰는 게 장고 admin 사이트(?)에서 db 볼 때 자동으로 막 반영되고 이런 게 신기했었는데<br>
spring에서는 안 되겠지 싶었다.<br>
저번에 jsp 프로젝트 할 때 datagrip을 쓰기도 했고 sql문도 직접 써서 그렇게 느꼈는데<br>
jpa를 통해 아 이것도 db 코드에서 sql문 없이 제어하는 게 있구나 싶어서 신기했다.<br>
장고 장점이 dbsqlite3인가 그거 제공해 주는 거라고 알고 있었는데<br>
사실 spring도 entity + jpa 쓰면 장고에 비해 단점이라고 느낄 정도는 아니라고 생각했다.<br>
오히려 장고는 깃허브 올리고 pull 받을 때마다 db 충돌이 계속 나서 불편했는데 springboot는 그럴 일 없으니까 개인적으로 이게 더 편한 느낌?<br>
물론 그때 깃허브도 잘 다룰 줄 모르고 장고도 뭣도 모르고 GPT랑 싸우기도 하고 급한 마음에 알아볼 생각 없이 넘어갔던 건 있다.<br>
그리고 django의 mtv 패턴이랑 spring boot의 mvc 패턴이 진짜 차이가 없다고 느꼈다.<br>
근데 우선 spring boot에서 비즈니스 로직이 훨씬 더 많이 분리된 느낌?
